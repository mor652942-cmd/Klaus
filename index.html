[ Sobre o projeto ]

Projeto Klaus


Klaus


Klaus é um sistema operacional escrito do zero, em Assembly x86, como projeto de aprendizado sobre como sistemas operacionais realmente funcionam por baixo dos panos — sem frameworks, sem bibliotecas, sem nada além do processador e da BIOS.


O que já funciona:

- Bootloader (boot/boot.asm) — o primeiro código a rodar na máquina, carregado pela BIOS. Lê o kernel e o shell do disco, habilita a linha A20, monta uma GDT e faz a virada de modo real (16 bits) para modo protegido (32 bits).

- Kernel (Kernel/kernel.asm) — carregado pelo bootloader, roda em modo protegido e repassa o controle para o shell.

- Shell (Shell/shell.asm) — a interface com quem usa o Klaus. Lê o teclado diretamente via I/O de baixo nível (sem depender de BIOS, já que em modo protegido essas chamadas não existem mais), ecoa o que é digitado na tela, mantém um prompt fixo (klaus>) e reconhece comandos.


Comandos disponíveis

Comando1: reiniciar > Reinicia a máquina de verdade, via reset de hardware pelo controlador de teclado.


Arquitetura

BIOS liga a máquina > boot.asm (setor de boot, 512 bytes, endereço 0x7C00) *lê do disco e carrega em memória* > kernel.asm (endereço 0x10000, modo protegido 32 bits) *passa o controle* > shell.asm (endereço 0x20000, lê teclado, interpreta comandos).


Cada estágio é compilado separadamente e depois concentrado em uma única imagem de disco (.img), na ordem:

boot → kernel → shell.


Como é compilado e testado

O projeto já vem com uma configuração para o VS Code: basta abrir a pasta no editor e apertar Ctrl+Shift+B. Isso monta os três estágios com o NASM, concentra tudo em uma imagem de disco e abre o resultado no QEMU automaticamente.


Testando em hardware real

O Klaus usa um bootloader clássico de setor MBR (BIOS Legada), então só funciona em máquinas com suporte a boot Legacy/CSM habilitado na BIOS. Notebooks modernos com UEFI puro (sem CSM) não conseguem bootar essa imagem sem suporte adicional.


Sobre

Projeto pessoal de aprendizado, construído passo a passo — cada peça (boot, modo protegido, kernel, leitura de disco, teclado, comandos) foi implementada e testada individualmente antes de avançar para a próxima.
